Evaluation

Final-round evaluation structure

The final thesis evaluation will freeze the deployed retrieval system, rerun a small controlled model matrix, and report both RAGAS diagnostics and human-facing review metrics.

Frozen systemBGE-M3 pgvector index

The final run should match the deployed PostgreSQL/pgvector retrieval system, not the older local Qwen embedding index.

Controlled comparisonSame evidence and prompt

Every model receives the same cached Top-5 evidence and the same final answer instruction.

Main claim boundaryFixed-context generation

The model sweep compares grounded answer synthesis from fixed evidence. Retrieval quality is diagnosed separately.

Protocol

Run order

This is the order we should follow before writing final claims. It prevents the experiment from mixing corpus changes, retrieval changes, prompt changes, and model changes in the same result table.

StepOutput artifactReason
1. Freeze retrievalretrieval_cache.jsonlEvery model receives the same Top-5 evidence.

RAGAS-native

Metrics we ask RAGAS to compute

These metrics are produced by the RAGAS framework. They remain diagnostic signals, not the full definition of user-facing answer quality.

MetricOriginMain inputsCalculation logic in plain language
faithfulnessRAGASquestion, generated answer, retrieved contextsThe evaluator decomposes the generated answer into factual claims and checks whether each claim is supported by the retrieved contexts. Higher means the answer is more grounded in evidence.
answer_relevancyRAGASquestion, generated answerThe evaluator checks whether the answer actually responds to the user question. RAGAS commonly does this by generating question-like probes from the answer and comparing them with the original question.
answer_correctnessRAGASquestion, generated answer, gold/reference answerThe evaluator compares the generated answer against the reference answer. It is intended to capture factual overlap and factual disagreement, not only wording similarity.
semantic_similarity / answer_similarityRAGASgenerated answer, gold/reference answer, embedding modelThe evaluator embeds the generated answer and reference answer, then computes semantic closeness in embedding space. This is wording-insensitive but can miss factual details.

Human review

Human-facing rubric

The thesis should use these metrics beside RAGAS because the same RAGAS score can hide differences in readability, usefulness, citation support, and unsafe over-answering.

MetricScaleHigher meansWhy it is needed
answer_acceptance0 / 1 / 2The answer is safer and more useful to show to a student.This is the north-star review signal for student-facing quality.
readability0 / 1 / 2The answer is clearer, more concise, and easier to follow.RAGAS can miss verbosity, repetition, awkward language, or mixed-language drift.
citation_usefulness0 / 1 / 2The citations support the key claims clearly.A cited answer is only useful if the citation actually points to supporting evidence.
over_answering_risk0 / 1 / 2Lower is better.This captures cases where the evidence is insufficient but the model still gives a specific answer.

Reference data

Gold answers and answerability

After the final retrieval cache is frozen, the gold answers and answerability labels must be regenerated or audited against that exact Top-5 evidence.

gold_answer is consumed by reference-based RAGAS metrics. is_answerable is project metadata for interpreting evidence sufficiency; it is not a native RAGAS metric.

Status

How to treat earlier results

The May 1 thinking/non-thinking run and earlier Gemma/Qwen runs are pilot evidence. They explain why we need a controlled final round, but they should not be used as the final headline result.

Final claims should come from the frozen retrieval cache, controlled prompt, small Qwen matrix, RAGAS scores, human-facing rubric, latency, output length, and failure tags.
