Evaluation method

How we use RAGAS

This page explains what RAGAS scores in our ICU Wiki experiments, which model performs each evaluation step, and which minimal project metadata is added for reference-based scoring.

Important distinctionJudge model ≠ tested model

Qwen or Gemma generates the answer. The RAGAS judge model evaluates that answer afterward.

Current judgeGPT-5 mini

The hard-20 benchmark uses openai/gpt-5-mini as the LLM evaluator through OpenRouter.

Current embeddingQwen embedding

The evaluator-side embedding model is Qwen/Qwen3-Embedding-0.6B.

Claim checking

How faithfulness is judged

For faithfulness, RAGAS does not simply compare two answer embeddings. It first uses the judge LLM to break the generated answer into factual statements. Then the judge checks whether each statement is supported by the retrieved context.

In our hard-20 run, that LLM-judged part is handled by openai/gpt-5-mini. The retrieved Top-5 evidence is the evidence base used for the support check.

Pipeline

From answer to faithfulness score

StepWhat happensWhy it matters
1. Answer statement extractionRAGAS asks the judge LLM to decompose the generated answer into smaller factual statements or claims.A long answer can contain both supported and unsupported claims. Claim-level scoring avoids treating the whole answer as one block.
2. Context support judgmentFor each claim, the judge LLM receives the retrieved context and decides whether the claim is supported by that context.This is where hallucinated or unsupported details should be penalized.
3. Claim aggregationThe supported-claim count is divided by the total number of claims.The final faithfulness score becomes a proportion, so partially grounded answers can receive intermediate scores.
4. Separate relevance and similarity checksOther metrics check different questions: whether the answer addresses the user question and whether it is close to the gold answer.This is why a model can be faithful but still incomplete, or semantically similar but not well cited.

Metric logic

Metric formulas

These formulas are intentionally simplified so the evaluation logic is explainable in a thesis presentation. The exact prompts and class names are handled by the installed RAGAS version.

MetricFormulaWhat the variables meanMain evaluator
faithfulness
S = {s1, ..., sn}
I(si) = 1 if claim si is supported by retrieved context, else 0
faithfulness = (1 / n) Σi=1n I(si)
S is the set of generated-answer claims. The score is the fraction of generated claims supported by retrieved evidence.Judge LLM for claim extraction and support checking.
answer_relevancy
Generate reverse questions q1, ..., qN from answer a
answer_relevancy = (1 / N) Σi=1N cos(E(q), E(qi))
cos(x, y) = (x · y) / (||x|| ||y||)
q is the original user question. qi are question-like probes generated from the model answer.Judge LLM creates probes; embedding model computes cosine similarity.
semantic_similarity / answer_similarity
semantic_similarity = cos(E(a), E(g))
cos(x, y) = (x · y) / (||x|| ||y||)
a is the generated answer. g is the gold/reference answer.Embedding model.
answer_correctness
precision = TP / (TP + FP)
recall = TP / (TP + FN)
F1 = 2 · precision · recall / (precision + recall)
answer_correctness ≈ wfactual · F1 + wsemantic · semantic_similarity
TP: generated claims present in the gold answer. FP: generated claims not present in gold. FN: gold claims missing from generated answer. RAGAS legacy defaults use weights 0.75 factual / 0.25 semantic.Judge LLM for factual comparison; embedding model for semantic part.

Evaluator roles

Where did we use GPT-5 Mini for?

GPT-5 mini is used as the evaluator LLM, not as the answer-generation model being tested.

It is used for LLM-judged steps: extracting claims, checking whether claims are supported by retrieved evidence, creating reverse questions for answer relevancy, comparing generated answers against gold answers, and generating the gold reference data.

The embedding-based parts are handled separately by Qwen/Qwen3-Embedding-0.6B. This includes semantic similarity, cosine-similarity calculations in answer relevancy, and the semantic-similarity component used inside answer correctness.

Reference data

What gold answers are for

The no-gold evaluation pass can score faithfulness and answer relevancy because those only need the question, model answer, and retrieved contexts.

To score answer correctness and semantic similarity, we need a reference answer. For that, GPT-5 mini generated gold_answer and is_answerable from the same fixed Top-5 evidence.

This gold-answer generation is our benchmark preparation step. It is related to RAGAS because RAGAS consumes the reference answer, but is_answerable is our addition.

Custom fields

It is used to interpret whether the fixed retrieved evidence can support a reference answer, but it can contain teacher-label errors.

Answerability example

How is_answerable is created

The label is produced before RAGAS scoring. We pass the question and fixed Top-5 retrieved evidence to GPT-5 mini. The model must decide whether the evidence is sufficient to write a grounded reference answer.

Prompt rule given to GPT-5 mini

This is the benchmark-preparation prompt that creates the answerability label. It tells GPT-5 mini to use only the fixed Top-5 retrieved evidence.

Implementation

Where this lives in the repository

code/eval/ragas_config.py selects the judge model, embedding model, and RAGAS metric objects.

code/eval/run_ragas_metrics.py builds the RAGAS dataset rows and calls ragas.evaluate(...).

code/eval/generate_gold_answers.py creates teacher-generated references and answerability labels before the with-gold scoring pass.
